# Wallet Compatibility

There are two distinct things a wallet can do for ZNS, and they have very different bars:

1. **Send a ZNS transaction** - broadcast a Zcash payment with a `ZNS:` memo so the indexer can pick it up. The wallet needs **zero** ZNS-specific code; it just has to handle the ZIP-321 URI the web app generates. Any modern shielded Zcash wallet that meets the three requirements below will work.
2. **Resolve a name in the wallet** - let the user type `alice.zcash` directly in the recipient field and have the wallet look up the underlying unified address before sending. This is per-wallet integration work and what "ZNS partnership" means.

## Sending: the three requirements

Any wallet that does these three things can broadcast a ZNS transaction generated by the web app:

1. **ZIP-321 payment URIs.** The app produces `zcash:<address>?amount=…&memo=…` URIs. Your wallet must parse these and prefill the send form.
2. **Orchard shielded transactions.** All ZNS memos travel in Orchard notes. Sapling-only wallets won't work.
3. **Verbatim memos.** The wallet must transmit the memo byte-for-byte - no normalization, no re-encoding, no whitespace trimming. The indexer verifies Ed25519 signatures over the exact memo bytes; a single-character change invalidates the action.

## In-wallet name resolution

Resolution lets a user type `alice.zcash` in the recipient field instead of pasting a 200-character unified address. The wallet calls the indexer's `resolve` RPC, substitutes the underlying address, and sends as normal - completely transparent to the user.

## Become a partner wallet

Adding ZNS resolution to a wallet is small. The integration is one RPC call (`resolve(query)`) before the wallet hands an address to its existing send flow. The full surface to integrate:

- A way to detect that input is a `.zcash` name (1–62 lowercase ASCII letters and digits, optional `.zcash` / `.zec` suffix). See [Name Validation](/docs/protocol/name-validation).
- A call to `resolve` against an indexer ([JSON-RPC Reference](/docs/indexer/rpc-reference) or use one of the [SDK ports](/docs/sdk) - TypeScript, Rust, Dart, Kotlin, Swift, Python, Go, React Native).
- Substitute the returned `address` into your existing send form and proceed normally.
- Optionally verify the returned signature against the pinned `admin_pubkey` ([Signature Scheme](/docs/protocol/signatures)) for trust-minimized resolution.

That's it - ZNS does not require changes to how the wallet builds or signs transactions. ## Wallets to avoid

- **zecwallet / zecwallet-cli** - the [official Zecwallet site](https://zecwallet.co) marks all Zecwallet projects as legacy and unmaintained.
- **Hardware wallets** - as of 2026 none of them support *sending to* shielded Zcash addresses, so they can't construct the Orchard memo at all.
- **Sapling-only wallets** - ZNS memos travel in Orchard notes; Sapling-only wallets don't see them.
- **Exchange wallets and most multichain wallets** - typically transparent-only and won't construct shielded outputs with attached memos.
- **Wallets that strip or normalize memos** - if you're unsure, send yourself a test transaction with a known memo and verify the hex matches. Even a single-byte change invalidates the Ed25519 signature.

## Reporting wallet issues

If you find a wallet that *should* work but doesn't, file an issue at [`zcashme/zcashnames`](https://github.com/zcashme/zcashnames/issues) with:

- Wallet name + version + platform
- The exact ZIP-321 URI the web app generated
- The memo the wallet actually sent (from the tx history, not the confirmation screen)
- The txid if the transaction was broadcast
